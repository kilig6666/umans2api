# umans2api 前端面板 Premium Control Center 设计

日期：2026-04-18

## 1. 目标

本次前端优化优先目标不是继续堆功能，而是把当前“能用的运维页”升级成“更像正式产品的控制中心”。

核心要求：

1. 更高级、更像产品
2. 保留现有运维效率
3. 不引入前端框架迁移
4. 优先重构现有 `templates/index.html`

## 2. 当前问题

当前控制面板已经具备功能，但存在以下问题：

1. 顶部信息层级偏平
2. 横向 tabs 导航像工具页，不像产品后台
3. 账号管理的新增、编辑、批量操作堆在同一层，视觉负担重
4. 保活、日志、运行态页面偏工程化，缺少产品化信息组织
5. 整体缺少一个“首屏就能建立信任感”的主视觉结构

## 3. 选定方案

采用方案 A：

**Premium Control Center**

关键词：

- 克制
- 专业
- 深色产品化
- 强层级
- 少卡片
- 控制台感

## 4. 视觉主张

### 4.1 Visual Thesis

像正式 AI 网关控制中心：冷静、克制、可信、专业。

首屏不做营销式 hero，而是做“高质量产品控制台首屏”：

- 一眼看出系统在线
- 一眼看出账号池状态
- 一眼能进入最重要的工作区

### 4.2 风格原则

遵循高质产品后台风格：

1. 先做布局，不先堆组件
2. 少卡片，避免整页碎片化
3. 深色但不过分炫技
4. 状态色只服务信息，不作为装饰
5. 一屏只强调一个主信息面

## 5. 信息结构

### 5.1 顶层结构

页面重构为三段：

1. 顶部产品状态栏
2. 左侧主导航
3. 右侧主工作区

整体结构：

```text
┌─────────────────────────────────────────────┐
│ Top Bar：品牌 / 在线状态 / 快捷动作          │
├───────────────┬─────────────────────────────┤
│ Left Nav      │ Main Workspace              │
│ 概览          │ 当前模块核心内容            │
│ 账号池        │                             │
│ 保活          │                             │
│ 请求日志      │                             │
│ 配置          │                             │
│ 调试          │                             │
└───────────────┴─────────────────────────────┘
```

### 5.2 导航结构

导航不再只是 tabs 平铺，而是产品化主导航：

- 概览
- 账号池
- 保活监控
- 请求日志
- 配置管理
- 聊天调试

运行状态并入“概览”或“配置”页下的辅助区，不单独占一个主导航优先级。

## 6. 页面设计

### 6.1 概览页

作为默认首页。

职责：

- 展示系统是否可用
- 展示账号池健康情况
- 展示保活与轮询的最新状态
- 给出最重要的下一步动作

布局建议：

1. 顶部主摘要
   - 服务状态
   - 活跃账号
   - healthy 数量
   - expiring 数量
   - inflight 数量
2. 中部双栏
   - 左：账号池健康概览
   - 右：最近请求 / 最近保活摘要
3. 下部动作区
   - 新增账号
   - 批量保活
   - 聊天调试

### 6.2 账号池页

改成“列表 + 详情”的工作台。

左侧：

- 账号列表
- 勾选批量操作
- 状态标签
- plan / expires / errors 摘要

右侧：

- 新增账号
- 编辑账号
- 单账号详细信息
- 单账号操作

这样能把“资源列表”和“编辑工作区”分离，提升高级感和操作清晰度。

### 6.3 保活页

不再只有一个日志框。

结构建议：

1. 保活策略概览
   - session check
   - 页面续期
   - chat fallback 是否开启
2. 保活状态表
   - 每个账号的 expires / 最近 check / 最近 keepalive / 最近错误
3. 操作区
   - 全量跑一轮
   - 单账号保活
   - 批量保活

### 6.4 请求日志页

升级成“筛选 + 列表 + 明细”。

至少完成：

- 上方筛选条
- 中间日志表
- 点击行后出现明细区或展开区

重点展示：

- 命中账号
- 客户端模型
- 上游模型
- 状态码
- 耗时
- 错误原因

### 6.5 配置页

从“表单堆叠”改成“设置页”。

按语义分组：

1. 鉴权设置
2. 调度设置
3. 保活设置

每组都有：

- 组标题
- 一句说明
- 相关字段

## 7. 视觉系统

### 7.1 色彩

保留深色主题，但更产品化：

- 背景做轻层次差
- 面板做柔和边界
- Accent 颜色只保留一个主蓝
- 危险/警告/正常状态色更克制

### 7.2 排版

字体层级明确：

- 页面标题强
- 模块标题稳
- 标签轻
- 数据值突出

### 7.3 容器

减少无意义卡片数量。

原则：

- 大区域用布局分区
- 真正需要独立语义时再用卡片
- 表格和日志优先成为工作区的一部分，而不是“又一个盒子”

## 8. 交互设计

### 8.1 动效原则

只加少量高价值动效：

1. 首屏状态块轻微入场
2. 左侧导航切换高亮过渡
3. 列表到详情面板的切换过渡

不做：

- 花哨渐变动画
- 大面积玻璃拟态
- 装饰性粒子或光效

### 8.2 工作流优化

关键操作要更接近真实产品体验：

- 点击账号进入右侧详情
- 批量操作固定在账号页工具栏
- 聊天调试区强调结果展示与最近命中账号

## 9. 技术实现边界

本次前端优化仅做：

1. `templates/index.html` 结构重排
2. CSS 体系升级
3. 少量原生 JS 交互重组

不做：

1. 引入 React/Vue
2. 改后端 API 语义
3. 大规模改数据结构

## 10. 实施顺序

### 第一批

1. 页面 shell 重构
2. 顶部状态栏升级
3. 左侧主导航替代横向 tabs
4. 概览页重建

### 第二批

5. 账号池页改成列表 + 详情布局
6. 保活页重构
7. 配置页重构

### 第三批

8. 日志页重构
9. 聊天调试区重构
10. 补动效与响应式细节

## 11. 验收标准

1. 首屏看起来像正式产品后台，而不是脚本工具页
2. 默认首页能在一屏内建立系统信任感
3. 账号管理区的结构明显更清晰
4. 保活、日志、配置的层级更清楚
5. 不影响现有核心功能使用
